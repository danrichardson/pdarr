## Description
The history tab accumulates completed jobs indefinitely. Users should be able to clear old "Done" entries from the history view to keep it manageable, but clearing history must NOT cause those files to be picked up again by the scanner and re-processed.

This is directly related to TKT-004 (files reappearing after being handled). The underlying requirement is that "processed" state must be tracked separately from "visible in history" — history is a UI concern, processing state is a data concern.

## Investigation
<!-- Agent fills this during /ticket-investigate -->

### How history clearing currently works

**API endpoint:** `POST /jobs/clear` handled by `handleClearHistory` in `internal/api/handlers.go:236`. It calls `db.ClearHistory()`.

**DB operation** (`internal/db/jobs.go:233`): Hard-DELETEs all job rows with terminal statuses `('failed','cancelled','skipped','excluded','restored')`. Also deletes associated `originals` rows first (FK cleanup). Notably, `done` and `staged` jobs are NOT deleted -- only error/skip/excluded states.

### Clearing deletes records entirely

There is no soft-delete, archive flag, or hidden column. `ClearHistory()` uses `DELETE FROM jobs WHERE status IN (...)`. Once deleted, the row is gone from the database permanently.

### How the scanner checks for already-processed files

`Scanner.maybeEnqueue()` (`internal/scanner/scanner.go:122`) calls `db.SourcePathStatus(path)` which runs:
```sql
SELECT status FROM jobs WHERE source_path=? ORDER BY created_at DESC LIMIT 1
```
If no row exists (returns empty string), the file is treated as never-seen and gets enqueued. The scanner only skips re-queuing when an existing job row has a non-retriable status (anything other than `""`, `failed`, or `restored`).

**This is the root cause of TKT-004:** When `ClearHistory()` deletes `excluded` job rows, the scanner's `SourcePathStatus` returns `""` for those files on the next scan, causing them to be re-queued. Files that were permanently excluded (uncompressible, repeated failures) lose their exclusion state entirely.

### No separation between processing state and history visibility

The `jobs` table serves dual purposes:
1. **Scanner state tracking** -- `SourcePathStatus()` queries it to decide whether to enqueue
2. **UI history display** -- `ListJobs()` queries it to populate the History tab

There is no independent "processed files" or "file state" table. Processing memory is entirely dependent on job rows existing.

### Relationship to TKT-004

TKT-004 reports files reappearing after processing. This ticket (TKT-006) is the direct mechanism: clearing history deletes the job records that the scanner relies on to know a file was already handled. The two tickets share the same root cause -- no separation of concerns between history display and processing state. Fixing TKT-006 properly will also fix TKT-004.

### Impact of current statuses deleted

| Status | Re-queue risk after delete | Severity |
|---|---|---|
| `excluded` | HIGH -- scanner will re-probe and re-queue these files | Critical |
| `skipped` | HIGH -- same as excluded | Critical |
| `cancelled` | MEDIUM -- user-cancelled files will be re-queued | Annoying |
| `failed` | LOW -- scanner already re-queues failed files anyway | Minimal |
| `restored` | LOW -- scanner already re-queues restored files | Minimal |

Regression Risk: medium (schema migration required; scanner logic and clear-history semantics both change; must not break existing job queries or the file browser's `GetJobMetaByPaths`)


## Proposed Solution
<!-- Agent fills this during /ticket-investigate -->

**Add a `hidden` boolean column to the `jobs` table** rather than deleting rows. This is simpler than a separate processed-files table and avoids duplicating data.

Approach:
1. Add `hidden INTEGER NOT NULL DEFAULT 0` column to the `jobs` table (via idempotent ALTER in `db.alterations()`).
2. Change `ClearHistory()` from `DELETE` to `UPDATE jobs SET hidden = 1 WHERE status IN (...)`. Remove the originals DELETE (originals rows should survive since job rows survive).
3. Update `ListJobs()` to filter `WHERE hidden = 0` by default so hidden jobs do not appear in the History tab.
4. `SourcePathStatus()` remains unchanged -- it still finds the job row because it was not deleted, so the scanner correctly skips already-handled files.
5. Add a `ClearHiddenJobs()` method (or extend the existing one) for permanent purge if needed later, but this is optional and can be deferred.
6. Update `GetJobMetaByPaths()` to continue returning all jobs (including hidden) so the file browser still shows status badges correctly.
7. Optionally, add a `hidden_at DATETIME` column for "clear older than X days" bulk filtering in the future.

**Why not a separate `processed_files` table?** A separate table would require inserting into it every time a job reaches a terminal state, and the scanner would need to query both tables. The `hidden` flag achieves the same goal with less disruption -- one new column, one changed query, zero new tables.
